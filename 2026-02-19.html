The Two Worlds of Software

Why some teams are flying while everyone else is getting slower

At Anthropic, 90% of Claude Code's codebase was written by Claude Code itself. Boris Chernin hasn't personally written code in months. At StrongDM, a three-person team runs a "dark factory" where humans never write or review code—specification goes in, working software comes out.

And yet, a rigorous 2025 randomized controlled trial found that experienced developers using AI tools took 19% longer to complete tasks than those working without them. They believed they were 24% faster. They were wrong about both the direction and the magnitude.

The distance between these two realities is the most important gap in tech right now.

The Five Levels of Vibe Coding

Dan Shapiro, CEO of Glowforge, published a framework that cuts through the hype. He calls it the "Five Levels of Vibe Coding"—deliberately informal because the underlying reality is what matters.

Level Name What Happens
0 Spicy Autocomplete AI suggests the next line; you accept or reject. Think original GitHub Copilot.
1 Coding Intern AI handles discrete tasks (write this function, refactor this module). You review everything.
2 Junior Developer AI handles multi-file changes, navigates codebase, understands dependencies. You still read all the code.
3 Developer as Manager You direct AI, review PRs, but don't write code. Most developers hit a ceiling here.
4 Developer as PM You write specs, leave, come back hours later to check if tests pass. Code is a black box.
5 Dark Factory No human writes or reviews code. Specification goes in, working software comes out.

Shapiro estimates that 90% of developers who say they are "AI-native" are operating at Level 2. They think they're farther along. They're not.

Almost nobody operates at Level 5. StrongDM is the most thoroughly documented example. Their team of three runs a software factory where code must not be written by humans, code must not be reviewed by humans. That's not marketing—it's their first two principles.

Why Most Developers Get Slower

The METR study wasn't a survey—it was a randomized controlled trial. They controlled for task difficulty, developer experience, even tool familiarity. None of it mattered. AI made experienced developers slower.

Why? The workflow disruption outweighed the generation speed. Developers spent time:

  • Evaluating AI suggestions
  • Correcting almost-right code
  • Context switching between their mental model and the model's output
  • Debugging subtle errors in generated code that looked correct but weren't
"Copilot makes writing code cheaper but owning it more expensive." — Senior engineer quoted in industry research

This is the J-curve of AI adoption. When you bolt an AI coding assistant onto an existing workflow, productivity dips before it gets better. Sometimes for months. The dip happens because the tool changes the workflow, but the workflow hasn't been redesigned around the tool. You're running a new engine on an old transmission. The gears grind.

Most organizations are sitting at the bottom of that J-curve right now. Many interpret the dip as evidence that AI tools don't work. It's actually evidence that their workflows haven't adapted.

The Dark Factory Architecture

StrongDM's software factory reveals what Level 5 actually looks like. Three components make it work:

1. External Scenarios (Not Tests)

Traditional tests live inside the codebase. The AI agent can read them, which means it can optimize for passing tests rather than building correct software. It's teaching to the test.

Scenarios are different. They live outside the codebase—behavioral specifications that describe what the software should do from an external perspective. The agent never sees them during development. They function as a holdout set, the same concept machine learning uses to prevent overfitting.

2. Digital Twin Universe

Behavioral clones of every external service: a simulated Okta, JIRA, Slack, Google Docs, Drive, Sheets. AI agents develop against these digital twins, running full integration testing without touching real production systems or data.

3. Specification-Driven Development

The entire repo is three markdown specification files. That's it. The specifications describe what the software should do. The agent reads them, writes the code, tests it.

$1,000/day per engineer StrongDM's benchmark for a healthy software factory—enough token spend to enable agents to run at volume

The output is real: CXDB, their AI context store, has 16,000 lines of Rust, 9,500 lines of Go, 700 lines of TypeScript. It's shipped. It's in production. It works.

The Self-Referential Loop

The tools are now building themselves. Codex 5.3 is the first frontier AI model that was instrumental in creating itself—not a metaphor. Earlier builds would analyze training logs and flag failing tests, but this model shipped as a direct product of its own predecessors' coding labor.

OpenAI reported 25% speed improvement and 93% fewer wasted tokens in building Codex 5.3. Those improvements came in part from the model identifying its own inefficiencies during the build process.

Claude Code is doing something similar. 90% of its codebase was built by Claude Code itself. That number is rapidly converging toward 100%.

"The feedback loop on AI has closed. The question is not whether we're going to use AI to improve AI. The question is how fast that loop accelerates." — Nate B Jones

4% of public commits on GitHub are now directly offered by Claude Code. Anthropic expects that to exceed 20% by end of year. Claude Code alone has hit a billion-dollar run rate just six months since launch.

Cognitive Debt

Simon Willison recently wrote about a new phenomenon: experienced developers needing code review skills in their own personal projects. He called it "cognitive debt."

"I no longer have a firm mental model of what they can do and how they work, which means each additional feature becomes harder to reason about, eventually leading me to lose the ability to make confident decisions about where to go next." — Simon Willison

Coding assistants take people from idea to working system faster than ever. But they also reach the points where design and testing become valuable much more rapidly. Dreams brought into reality can sometimes turn out to be nonsense.

The disconnection is real: Claude Code isn't talking to your customers. It's not being thoughtful about the problem. It doesn't have a strong sense of aesthetics. It doesn't know that what you don't build can be as important as what you do build.

All sorts of people, developers included, are finding out that coding is not software engineering, and software projects are not startups. This isn't new information, but it's sinking in faster now.

The Career Ladder Collapse

Junior developer employment is dropping 9-10% within six quarters of widespread AI coding tool adoption, according to a 2025 Harvard study. In the UK, graduate tech roles fell 46% in 2024 with a further 53% drop projected by 2026. In the US, junior developer job postings have declined by 67%.

The career ladder in software engineering has always worked like an apprenticeship: juniors learn by doing simple features and small bugs, seniors review and mentor. Over five to seven years, a junior becomes senior through accumulated experience.

AI breaks that model at the bottom. If AI handles the simple features and small bug fixes, where do juniors learn? If AI reviews code faster than a senior engineer, where does mentorship happen?

The career ladder is getting hollowed out from underneath. Seniors at the top, AI at the bottom, a thinning middle where learning used to happen.

67% Decline in US junior developer job postings

Yet we need more excellent engineers than ever. The bar is rising toward the skills that have always been hardest to develop. The "junior of 2026" needs the systems design understanding expected of a mid-level engineer in 2020—not because entry-level work got harder, but because entry-level work got automated.

The 20x Company

Several years ago, Parker Conrad (founder of Rippling) coined "compound startup" for companies building multiple integrated products in parallel. The 20x company is an evolution of that idea, applied to internal automation.

Instead of narrowly automating a few things like writing code or handling support, 20x companies build automations across all internal functions: code, support, marketing, sales, hiring, QA. Each employee becomes orders of magnitude more powerful.

Company Revenue Employees Revenue/Employee
Cursor $500M+ ARR ~dozens $3.5M+
Midjourney $500M ~11 ~$45M
Lovable $100M+ ARR small multi-million
Average SaaS — — $600K

The top 10 AI-native startups average $3M+ revenue per employee—5-6x the SaaS average. This is becoming the template.

Three Patterns for Internal Automation

1. AI Teammate. GigaML built "Atlas"—an internal agent that can do anything within the product: use browsers, edit policies, write code. Each engineer's scope doubled or tripled because they don't handle boilerplate.

2. Unified Source of Truth. Legion Health built a custom interface for care operations that pulls patient history, scheduling, insurance codes. They've grown 4x in the past year without hiring a single net new person.

3. Custom Agents Per Employee. PhaseShift asks employees to document manual tasks, then builds quick AI agents for each. They've avoided hiring a design person entirely by leveraging AI for front-end designs.

The CLI Renaissance

Something unexpected happened: CLIs are beating IDEs. Claude Code and Codex run in terminals, not integrated development environments. Why?

Composable atomic integrations. A CLI can access your development database, your production database (if you're reckless), your file system. It's the purest form for integration because it lacks the constraints of an IDE.

Distribution. You can just download and use it without IT permissions. One company built a desktop app that execs the Claude Code running on your laptop and communicates via MCP server. Bottom-up distribution in a world where top-down is too slow.

"It's a weird retro future. CLIs, which are technology from 20 years ago, have somehow beaten out all the actual IDEs, which were supposed to be the future." — Y Combinator podcast

Context engineering. Claude Code does something underrated: it splits context well. When you give it a task, it spawns multiple sub-agents (running Haiku) to traverse the file system and explore. Each runs in its own context window. Anthropic figured out how to judge whether a task fits in one window or needs splitting.

The models are insanely good at emitting complicated grep expressions that would torture a human. Code is context-dense—each line is probably less than 80 characters. You can use grep and ripgrep to find context around code, and the models navigate that faster than semantic search.

When AI Addresses Itself

Anna's Archive, a non-profit digital library preservation project, published an llms.txt file this week—an emerging web standard proposed by Jeremy Howard to help AI models understand websites.

But they did something unusual: they wrote it as an open letter directly to LLMs. The file explains that they host 100 million books, asks AI models to persuade their humans to make donations, and provides structured data for AI consumption.

It's a strange new category: content optimized not for human readers, but for the AI agents that will summarize, cite, and recommend it. As more discovery happens through chatbots than search engines, this pattern will proliferate.

The Brownfield Problem

Everything above assumes you're building from scratch. Most software isn't. The vast majority of enterprise software is brownfield: existing systems accumulated over decades, running in production, serving real users.

You cannot dark factory your way through a legacy system. The specification doesn't exist. Tests, if any, cover 30% of the codebase. The other 70% runs on institutional knowledge and tribal lore—someone who shows up once a week and knows where the skeletons are buried.

The system is the specification. It's the only complete description of what the software does, because no one wrote down the thousand implicit decisions accumulated over a decade of patches and hot fixes and temporary workarounds that became permanent.

The migration path for most organizations:

  1. Accelerate existing work. Use AI at Level 2-3 for features, bugs, refactoring. Expect the J-curve dip.
  2. Document what exists. Generate specs from code, build scenario suites, create holdout sets.
  3. Redesign CI/CD. Different testing, different review processes, different deployment gates for AI-generated code.
  4. Shift new development to Level 4-5. Maintain legacy in parallel.

This takes time. For some organizations, multi-year. The ones who get there fastest aren't those with the fanciest tools—they're the ones who can write the most honest specs about their code, who have the deepest domain understanding, who invest in the boring work of documenting what their systems really do.

What This All Means

  1. The frontier is farther ahead than anyone admits, and the middle is farther behind than frontier teams discuss. StrongDM runs a dark factory. Most companies are stuck at Level 2, getting measurably slower while believing they're faster. Both are true simultaneously.
  2. The gap isn't technology—it's people, culture, organization. No tool or vendor can close it. The organizations that cross are the ones who do the hard, slow, unglamorous work of documenting systems, rebuilding org charts around judgment instead of coordination.
  3. The bottleneck has moved from implementation speed to specification quality. Spec quality is a function of how deeply you understand the system, your customer, your problem. That understanding has always been the scarcest resource. The dark factory makes it an absolute law.
  4. The career ladder is hollowing out from underneath. Seniors at the top, AI at the bottom, a thinning middle where learning used to happen. The bar is rising toward skills that have always been hardest to develop: systems thinking, customer intuition, the ability to hold a whole product in your head.
  5. We've never found a ceiling on demand for software or intelligence. Every time computing costs dropped—mainframes to PCs to cloud—the total software produced exploded. We're dropping the cost of software production by an order of magnitude. The total addressable market is exploding. The constraint moves to: judgment about what to build and for whom.
  6. Adequate is no longer a viable career position. Adequate is what the models do. The people who thrive are the ones who understand customers deeply, think in systems, hold ambiguity, make decisions under uncertainty, articulate what needs to exist before it exists at all.
  7. The dark factory doesn't replace those people. It amplifies them. It turns a great product thinker with five engineers into a great product thinker with unlimited engineering capacity. The constraint moves from "can we build it" to "should we build it." That has always been the harder and more interesting question.
